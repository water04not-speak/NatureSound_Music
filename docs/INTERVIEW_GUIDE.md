# Interview Guide

这份文档用于面试前快速复盘项目。建议你按“项目背景 -> 我的职责 -> 技术难点 -> 解决方案 -> 结果与优化”来讲。

## 30 秒项目介绍

“万物之声”是一款基于 HarmonyOS ArkTS 开发的音乐播放应用。我主要实现了在线音乐播放、播放列表、歌词解析、收藏搜索、非遗音乐内容展示、互动社区、登录注册、个人中心、主题换肤和本地 RDB 数据持久化。项目中我把 `AVPlayer` 封装成播放器桥接层，把歌曲数据访问封装成 Repository 层，让页面层更关注交互和展示。

## 简历描述

基于 HarmonyOS ArkTS 独立开发音乐播放应用“万物之声”，实现在线音乐播放、后台播放、歌词解析、歌单推荐、非遗音乐内容展示、互动社区、用户登录注册、主题换肤和本地 RDB 数据持久化。项目封装 `AVPlayer` 播放器桥接层与 `MusicRepository` 数据访问层，支持播放状态同步、播放列表管理、收藏搜索、离线文件优先播放和远程流回退，具备较完整的移动端音乐 App 业务闭环。

## 面试讲解重点

### 1. 为什么封装播放器

不要说“为了方便调用”就结束。可以这样讲：

播放状态机比较复杂，页面直接操作 `AVPlayer` 会导致播放、暂停、切歌、进度、列表和后台状态混在一起。所以我把播放器逻辑收敛到 `AvPlayerBridge`，页面只调用 `singPlay`、`nextPlay`、`paused` 这类业务动作，降低页面耦合，也方便后续替换底层播放实现。

### 2. 如何处理播放失败

可以讲两个点：

- 部分音乐外链是 HTTP，项目里统一做了 HTTPS 规范化。
- 如果歌曲已下载，优先播放本地文件；本地文件播放失败时回退远程流，避免用户直接听不了。

### 3. 为什么使用 Repository

页面不应该直接写 SQL。`MusicRepository` 把搜索、收藏、种子数据同步、增删改查封装起来，页面只关心拿到什么歌曲列表。这样后续从本地 RDB 切换到后端 API，也可以尽量少改页面。

### 4. 项目差异化在哪里

普通音乐播放器很多，这个项目增加了“非遗音乐”内容模块，将地域音乐和传统曲艺做成内容入口。面试时可以强调它有明确产品主题，不只是为了实现播放器功能。

### 5. 如果继续做会怎么优化

- 接入真实后端，支持用户、评论、动态和歌单云同步。
- 增加播放历史、最近常听和推荐算法。
- 为播放器状态机、歌词解析和 Repository 增加自动化测试。
- 完善离线下载队列、缓存大小统计和失败重试。
- 优化平板布局、深色模式和无障碍适配。

## 高频问题准备

| 问题 | 回答方向 |
| --- | --- |
| 你项目里最难的地方是什么 | 播放器状态同步、后台播放、播放列表和本地/远程播放回退 |
| 为什么不用页面直接调用数据库 | 通过 Repository 解耦 UI 与数据访问，便于维护和替换数据源 |
| 项目怎么体现工程能力 | 有页面层、模型层、服务层、资源层，播放和数据都做了封装 |
| 你负责了哪些内容 | 可以说核心页面、播放器封装、数据持久化、歌词和主题换肤等 |
| 有什么不足 | 后端是假数据/本地种子，测试覆盖还可以继续补充 |

## GitHub 展示建议

- README 第一屏要能看到项目定位、技术栈和截图。
- 截图要展示完整业务闭环：启动、首页、播放、歌词、歌单、非遗、社区、登录、个人中心、设置。
- 不提交本机签名、构建缓存、压缩包和临时文件。
- 代码里保留有价值的封装，不把所有逻辑写在页面里。
